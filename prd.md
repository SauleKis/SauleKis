# PRD: «Берег» — подбор тура (5★, первая линия) по сезону и бюджету

## 1. Описание проекта

**Название:** Берег (рабочее)
**Тип:** личный веб-проект, прототип
**Один абзац:** Веб-приложение, которое помогает быстро подобрать отель уровня 5★ на первой линии от моря, исходя из времени года поездки и допустимого бюджета за ночь. Решает проблему долгого ручного перебора вариантов на сайтах-агрегаторах, когда у пользователя жёсткие требования по категории и расположению, но при этом важно не переплатить.

## 2. Проблема и цель

**Проблема:** Подбор тура с условием «5 звёзд + первая линия + недорого» занимает много времени, потому что эти три критерия редко фильтруются одновременно в готовом виде, а понятие «недорого» сильно зависит от сезона (один и тот же отель может стоить вдвое дороже в высокий сезон).

**Цель проекта:** Дать пользователю интерфейс, где за два клика (сезон + бюджет) виден отсортированный по цене список подходящих отелей, без необходимости вручную сверять сезонность и читать описания каждого варианта.

**Гипотеза:** Если зафиксировать категорию и расположение как обязательные условия и оставить пользователю только два «рычага» — сезон и бюджет, то результат будет восприниматься как релевантный и экономящий время.

## 3. Целевой пользователь

- Один пользователь (сам автор проекта) или небольшой круг знакомых.
- Человек, который уже определился с требованиями к отелю (5★, первая линия) и хочет понять, **когда и за сколько** в такой отель реально попасть.
- Не профессиональный путешественник, не готов разбираться в сложных фильтрах турагрегаторов.

## 4. Объём (Scope)

### 4.1 Входит в MVP (уже реализовано в прототипе)
- Фиксированные условия: категория отеля = 5★, расположение = первая линия (заложены как обязательные фильтры, не как опция для пользователя).
- Выбор времени года: зима / весна / лето / осень / любое.
- Слайдер бюджета (цена за ночь, $).
- Выбор направления (опционально, не обязательный фильтр).
- Сезонная логика цены: для каждого отеля задан «высокий» сезон; если выбранный сезон совпадает с высоким — показывается высокая цена, иначе — низкая (межсезонье).
- Сортировка результатов от дешёвых к дорогим.
- Пустое состояние, если ничего не подошло (с подсказкой увеличить бюджет или сменить сезон).
- Тестовая база из 15 отелей (статичные данные в коде).

### 4.2 Не входит в MVP (но рассматривается дальше — см. раздел 8)
- Реальные данные от поставщиков туров / отелей (API или парсинг).
- Бэкенд и база данных (сейчас всё хранится в JS-массиве на клиенте).
- Авторизация пользователей, сохранение избранного.
- Бронирование внутри приложения.
- Карта с расположением отелей.
- Мультивалютность и мультиязычность.

## 5. Пользовательские сценарии (User Flow)

**Основной сценарий:**
1. Пользователь открывает сайт.
2. Видит заголовок и краткое объяснение, что сайт ищет только 5★ отели на первой линии.
3. Выбирает время года (или оставляет «Любое»).
4. Двигает слайдер бюджета до комфортной суммы за ночь.
5. (Опционально) выбирает конкретное направление.
6. Получает список карточек отелей, отсортированный по возрастанию цены.
7. Если результатов нет — видит понятную причину и совет, что изменить.

**Альтернативный сценарий (пустой результат):**
- Пользователь поставил слишком низкий бюджет → видит empty state → увеличивает бюджет через слайдер → список появляется без перезагрузки страницы.

## 6. Функциональные требования

| № | Требование | Статус в прототипе |
|---|---|---|
| F1 | Показывать только отели с 5 звёздами | Реализовано (фильтр в коде) |
| F2 | Показывать только отели первой линии | Заложено как условие данных; явного поля `line` пока нет — все отели в тестовой базе считаются первой линией |
| F3 | Фильтр по сезону (5 значений, включая «любое») | Реализовано |
| F4 | Фильтр по бюджету (слайдер, $60–$600 за ночь) | Реализовано |
| F5 | Фильтр по направлению | Реализовано (необязательный) |
| F6 | Сезонная корректировка цены (высокий/низкий сезон) | Реализовано |
| F7 | Сортировка результатов по цене (возрастание) | Реализовано |
| F8 | Счётчик найденных вариантов | Реализовано |
| F9 | Пустое состояние с рекомендацией | Реализовано |
| F10 | Адаптивная вёрстка (мобильные устройства) | Реализовано |

## 7. Нефункциональные требования

- Прототип должен открываться как один HTML-файл без сборки и сервера.
- Время отклика фильтров — мгновенное (вся логика на клиенте, без сетевых запросов).
- Дизайн — тематика «море/побережье», читаемая типографика, доступность фокуса с клавиатуры.
- Код должен оставаться достаточно простым, чтобы один человек мог расширять его без рефакторинга всей архитектуры.

## 8. Дальнейшее развитие (Roadmap)

**Шаг 1 — данные:**
Добавить полю `line` (1/2/3-я линия) явное значение в структуре данных вместо предположения, что вся база — первая линия. Это нужно, если база вырастет и в неё попадут отели не первой линии.

**Шаг 2 — backend:**
Перенести список отелей из JS-массива в JSON-файл или базу данных (SQLite для личного проекта), поднять простой backend (Flask/FastAPI или Node/Express) с одним эндпоинтом фильтрации.

**Шаг 3 — реальные данные:**
Рассмотреть периодическое обновление цен (скрипт, который раз в день/неделю обновляет JSON) — без юридических рисков парсинга закрытых API крупных агрегаторов.

**Шаг 4 — UX-улучшения:**
- Карта с отелями (через интеграцию геосервиса).
- Сохранение избранных вариантов (localStorage или аккаунт).
- Расширенный фильтр: число ночей, тип питания.

## 9. Метрики успеха (для личного проекта)

Поскольку это не коммерческий продукт, метрики качественные:
- Автор находит подходящий вариант быстрее, чем при ручном поиске на агрегаторах.
- Интерфейс не требует объяснений — понятен с первого взгляда.
- Прототип легко расширяется новыми отелями без правок логики (достаточно добавить объект в массив/файл данных).

## 10. Риски и ограничения

- **Тестовые данные:** цены и сезонность отелей в MVP условные, не отражают реальный рынок — нужно явно маркировать это в интерфейсе (уже сделано в футере).
- **Понятие «недорого» субъективно:** решено через ручной слайдер бюджета, а не через автоматическое определение «дорого/дёшево» — это осознанный компромисс для простоты.
- **Источник данных в будущем:** если переходить на реальные данные, нужно заранее проверить условия использования API/сайтов, чтобы не нарушать их правила (без парсинга закрытых источников).
